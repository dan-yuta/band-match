監視の全体像（4層アーキテクチャ）

複数レイヤーで障害を検知し、重要度に応じて適切なチャネルに通知

🌐 Layer 1: 外形監視（Synthetic Monitoring）
CloudWatch Synthetics Canary — ユーザー視点でサービスの正常性を1分間隔で確認
↓
⚙️ Layer 2: インフラ監視（CloudWatch Metrics）
ECS CPU/メモリ、RDS コネクション、ALB 5xxエラー率、Redis キャッシュヒット率
↓
📝 Layer 3: アプリケーション監視（Logs + Sentry）
構造化ログ → Metric Filter でエラーパターン自動検出 / Sentry でフロントエンドエラー収集
↓
📊 Layer 4: ビジネスメトリクス監視
日次チェック: 新規登録数・ログイン成功率・決済成功率が前週比50%以下で異常検知
↓ 集約
🔔 通知・エスカレーション
Critical/High → Slack + メール + LINE ｜ Medium → Slack ｜ Low → 日次サマリー

障害対応ランブック（主要5パターン）

頻出障害の対応手順を事前定義

🔴 ECSタスク停止
  1. 停止理由を確認: aws ecs describe-tasks
  2. OOM → メモリ増量してデプロイ
  3. ヘルスチェック失敗 → DB/Redis接続確認
  4. イメージPull失敗 → ECRイメージ存在確認
🔴 RDS接続障害
  1. RDSステータス確認: aws rds describe-db-instances
  2. コネクション数確認 → 上限なら不要な接続をkill
  3. RDS停止 → 手動再起動: aws rds reboot-db-instance
  4. データ復旧 → 最新スナップショットからリストア
🟠 決済障害（Stripe）
  1. Stripeダッシュボード → Events でエラー確認
  2. 署名エラー → Secrets ManagerのWEBHOOK_SECRET確認
  3. Stripe側障害 → status.stripe.com 確認
  4. 失敗イベント → Stripeダッシュボードから手動リトライ
🟠 CDN障害（CloudFront）
  1. CloudFrontエラー率確認（CloudWatch）
  2. オリジン（ALB）の稼働確認
  3. キャッシュ起因 → Invalidation実行
  4. SSL証明書期限切れ → ACMで更新
🟡 WebSocket障害（チャット停止）
  1. API Gateway WebSocketのステータス確認
  2. Lambda実行エラーのログ確認
  3. DB接続の確認（チャット関連テーブル）
  4. 一時対応: API Gatewayステージを再デプロイ
