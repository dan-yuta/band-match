Section 01
🤔 なぜマルチアカウント？

シングルアカウントの問題点を明確にし、マルチアカウント設計がもたらす具体的なメリットを整理します。

⚠️ シングルアカウント（現状の問題点）
  • 開発・本番が同一アカウントで混在し、誤操作のリスクが高い
  • IAMポリシーが複雑化し、最小権限の原則を維持しにくい
  • コスト配分が困難で、環境別の費用が見えにくい
  • セキュリティインシデント発生時の影響範囲が全環境に及ぶ
  • コンプライアンス要件（本番データ分離）の対応が困難
  • リソース名のプレフィックスによる環境区別が人的ミスを誘発
  • サービスクォータ（制限値）を全環境で共有するリスク
→
✅ マルチアカウント（目指す姿）
  • アカウント境界によるハードな環境分離で誤操作を防止
  • 環境ごとにシンプルなIAMポリシー設計が可能
  • Consolidated Billingで環境別コストが明確に把握できる
  • 爆発半径（Blast Radius）を特定アカウントに限定
  • 本番アカウントへの厳格なSCPでコンプライアンスを担保
  • アカウント名で環境が自明となり命名規則が簡素化
  • 環境ごとにサービスクォータを独立管理
💥
爆発半径の隔離
本番環境での障害・セキュリティインシデントを他環境へ波及させない。アカウント境界が最強のガードレール。
📊
請求の分離
Consolidated Billingで一元管理しながら、環境別・チーム別のコストを正確に追跡・配分できる。
🔑
IAMの簡素化
各アカウント内のリソースはすべて同一環境のものため、IAMポリシーのスコープが明確になり管理が容易。
📋
コンプライアンス準備
本番アカウントにSCPで強制ガードレールを設定し、将来のSOC2/ISO27001対応の基盤を今から構築。

Section 03
🌳 AWS Organizations 構成

AWS Organizations で階層的なOU（Organizational Unit）構成を構築し、SCPによるポリシー強制を実現します。

🏢 Root（組織ルート）
🏛️ Management OU
👑 groove-management
ルートアカウント
SCPなし（管理用）
⚙️ Workloads OU
🛠️ groove-dev
🧪 groove-staging
🚀 groove-production
SCP: DenyVPCDeletion, DenyRDSDeletion, RegionRestrict
📋 SCP 継承ルール
親OUのSCPは子アカウントに自動継承 → Workloads OUのSCPは3アカウント全てに適用
🔒
DenyVPCDeletion
VPC削除の禁止（全Workloads）
// SCP: DenyVPCDeletion
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyVPCDeletion",
    "Effect": "Deny",
    "Action": [
      "ec2:DeleteVpc"
    ],
    "Resource": "*"
  }]
}
🛡️
DenyRDSDeletion
RDS削除保護（本番アカウント）
// SCP: DenyRDSDeletion
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyRDSDeletion",
    "Effect": "Deny",
    "Action": [
      "rds:DeleteDBInstance",
      "rds:DeleteDBCluster"
    ],
    "Resource": "*"
  }]
}
🌏
RegionRestriction
ap-northeast-1 のみ許可
// SCP: RegionRestriction
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyNonApNE1",
    "Effect": "Deny",
    "Action": "*",
    "Resource": "*",
    "Condition": {
      "StringNotEquals": {
        "aws:RequestedRegion":
          "ap-northeast-1"
      }
    }
  }]
}
💡 SCPの重要な特性： SCPは「許可」ではなく「最大許可範囲の制限」です。SCPで許可してもIAMで拒否されれば操作できません。またSCPで拒否した操作はAdministratorAccessを持つユーザーでも実行不可となります。

Section 05
🛡️ 本番環境 多層保護策

本番環境への変更・削除操作を複数のレイヤーで防御します。単一の防御が突破されても他のレイヤーが守ります。

1
🏛️
SCP（Service Control Policy）
Organizations レベルでVPC/RDS削除を完全禁止。AdministratorAccessでも突破不可。
Organizationsレベル
2
🔐
IAM 最小権限原則
本番アカウントへの人間のアクセスはReadOnlyのみ。デプロイはCI/CD専用OIDC Roleのみ。
IAMレベル
3
📦
CloudFormation Deletion Protection
スタックの削除保護を有効化。Termination Protectionが有効な間はスタック削除コマンドが失敗する。
CloudFormationレベル
4
📋
CloudFormation Stack Policy
スタックポリシーで特定リソース（RDS, VPC）への更新・置換・削除操作を明示的に拒否。
スタックポリシー
5
🗄️
RDS 削除保護 ＋ 最終スナップショット
DeletionProtection: true 設定。削除試行時は先にDeletionProtectionを無効化する操作が必要で、誤削除を防止。
RDSレベル
6
✅
本番デプロイ承認フロー
GitHub Actionsの手動承認ステップ（Environment Protection Rules）。指定レビュアーの承認がないと本番デプロイが実行されない。
CI/CDレベル
📦
CloudFormation Termination Protection
全本番スタックに設定
# CloudFormation スタック定義例
AWSTemplateFormatVersion: '2010-09-09'
Description: Groove Match Prod Stack

Resources:
  ProductionDatabase:
    Type: AWS::RDS::DBCluster
    DeletionPolicy: Snapshot
    UpdateReplacePolicy: Snapshot
    Properties:
      DeletionProtection: true
      BackupRetentionPeriod: 7
📋
CloudFormation Stack Policy
重要リソースの変更を拒否
// Stack Policy JSON
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "Update:Modify",
      "Principal": "*",
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action":
        ["Update:Replace",
         "Update:Delete"],
      "Principal": "*",
      "Resource":
        "LogicalResourceId/DB*"
    }
  ]
}

Section 08
📅 移行手順・タイムライン

シングルアカウント構成からマルチアカウント構成への移行を6週間で完了します。サービス停止ゼロを目標とします。

🎯 移行目標： 本番サービスのダウンタイム ゼロ。ブルーグリーンデプロイメントを活用し、新旧環境を並行稼働させながら段階的に移行します。
1
Week 1
🏛️ AWS Organizations 作成・設定
管理アカウントでOrganizationsを有効化し、OU構造を作成。SCPの基本ポリシーを定義・適用する。
Organizations有効化 Management OU作成 Workloads OU作成 SCPポリシー定義 Consolidated Billing設定
2
Week 2
🏦 アカウント作成・基本設定
groove-dev, groove-staging, groove-production の3アカウントを作成し、各OUに配置。基本セキュリティ設定を適用。
Dev アカウント作成 Staging アカウント作成 Prod アカウント作成 CloudTrail有効化 SecurityHub有効化 Config有効化
3
Week 3
🛡️ SCP 適用・IAM Identity Center設定
SCP ポリシーをWorkloads OUに適用。IAM Identity Centerを管理アカウントで有効化し、Permission Setを定義・割り当て。
SCP適用・テスト IAM Identity Center有効化 Permission Set作成 ユーザー/グループ割り当て MFA強制設定

🔗 GitHub OIDC・CI/CD設定

Week 5
📦 CloudFormation スタック展開
各アカウントにCloudFormationスタックを順次展開。Dev→Staging→Prodの順で実施し、各環境での動作確認後に次へ進む。
VPCスタック（Dev） VPCスタック（Stg） VPCスタック（Prod） ECSクラスタ展開 RDS構築 動作確認・テスト
