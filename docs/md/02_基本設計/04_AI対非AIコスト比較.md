# Groove Match - AI対非AIコスト比較

AIマッチングを導入すべきか、現行の非AIスコアリングを維持すべきか。本ドキュメントでは、現行アルゴリズムの評価、AIアプローチの候補と費用試算、段階的な導入ロードマップ、ROI分析を整理し、意思決定の根拠を示します。

---

## 1. 現行アルゴリズム評価

### 1.1 現行アルゴリズム概要

現行は6要素によるルールベーススコアリングです。

```
総合スコア = 地域(40) + 楽器(20) + ジャンル(20) + スキル(15) + スケジュール(30) + 曲(35)
           = 最大160pt → 100pt上限でクランプ
```

曲データはSpotify APIで検索しSpotify IDのみをDBに保存。マッチング時はIDを直接比較するため、API呼び出しコストは検索時のみで、マッチング計算自体は無料です。

### 1.2 強み

| 強み | 詳細 |
|------|------|
| **低コスト** | マッチング計算は純粋なSQL + Redisキャッシュで完結。外部API費用ゼロ |
| **決定論的** | 同じ入力に対して常に同じスコアが得られる。デバッグ・QAが容易 |
| **説明可能性** | 「曲が2曲一致したため+14pt」のようにスコアの内訳をユーザーに提示できる |
| **低レイテンシ** | RedisキャッシュとDB索引を活用すれば100ms以内でマッチング結果を返せる |
| **コピバン特化** | 曲スコア35ptはコピバンという特殊なユースケースに最適化されており、汎用AIより適合する可能性がある |
| **インフラ不要** | 追加のML基盤・GPUインスタンス・外部モデルAPIが不要 |

### 1.3 弱み

| 弱み | 詳細 | 影響度 |
|------|------|--------|
| **深い好みの不理解** | 「BUMPが好き」「初心者だけど上達意欲が高い」といった文脈をスコアに反映できない | 中〜高 |
| **コールドスタート問題** | 登録直後でプロフィールが薄いユーザーはスコアが低くなり、マッチングが成立しにくい | 高 |
| **行動からの学習なし** | 「プロフィールを見た」「チャットを開始した」などの行動データをスコアに反映できない | 中 |
| **要素の重み付けが固定** | 地域40pt・曲35ptという重みは経験則による設定。ユーザー属性によって最適な重みは異なる可能性がある | 低〜中 |
| **曲未登録ユーザーへの対応** | やりたい曲を登録していないユーザーは曲スコア0pt。コピバン特化であるにもかかわらずマッチング精度が大幅に低下する | 高 |

## 2. AI候補アプローチ

### 2.1 Embedding（埋め込みベクトル）

#### 概要

ユーザープロフィール（好きなジャンル・アーティスト・曲・自己紹介文）をベクトルに変換し、コサイン距離で類似度を算出するアプローチです。

| 項目 | 内容 |
|------|------|
| **モデル** | `text-embedding-3-small`（OpenAI、$0.020/1Mトークン）または music2vec（OSS） |
| **計算タイミング** | プロフィール更新時にベクトルを生成・保存（オフライン） |
| **マッチング時** | pgvector（PostgreSQLの拡張）でコサイン類似度検索 |
| **ストレージ** | 1ユーザー = 1,536次元のfloat32 = 約6KB |

#### 費用試算（text-embedding-3-small）

プロフィール1件あたり約500トークンと想定。

| MAU | プロフィール更新頻度（月） | 月間トークン | 月額API費用 | 円換算（¥150/ドル） |
|-----|--------------------------|------------|-----------|------------------|
| 1,000 | 2,000件 | 1Mトークン | $0.02 | **約¥3** |
| 5,000 | 10,000件 | 5Mトークン | $0.10 | **約¥15** |
| 20,000 | 40,000件 | 20Mトークン | $0.40 | **約¥60** |

> pgvectorをRDS PostgreSQLに追加する場合、インスタンスクラスをdb.t4g.microからdb.t4g.smallへアップグレードが必要になる可能性があります（月額+¥3,000〜5,000）。

#### 特徴

- **強み**: テキスト（自己紹介文・ジャンル名）の意味的類似度を捉えられる。「ロック」と「邦ロック」の近さを理解できる
- **弱み**: 曲IDの一致（現行の強み）をそのままでは扱えない。music2vecはコピバン文脈での学習データが乏しい
- **導入コスト**: 中程度（pgvector設定 + 埋め込み生成バッチ処理の実装）

---

### 2.2 LLM推薦（GPT-4o-mini）

#### 概要

ユーザーのプロフィールを自然言語として解釈し、LLMがマッチング候補をスコアリング・説明するアプローチです。

| 項目 | 内容 |
|------|------|
| **モデル** | `gpt-4o-mini`（$0.150/1Mトークン入力、$0.600/1Mトークン出力） |
| **用途** | マッチング理由の自然言語説明、コールドスタートユーザーへの推薦文生成 |
| **呼び出しタイミング** | マッチング画面表示時（オンライン） または 夜間バッチ（オフライン推奨） |

#### 費用試算（マッチング説明文生成：1回あたり入力1,000トークン・出力200トークン）

| MAU | 月間マッチング表示回数 | 月額API費用 | 円換算 |
|-----|---------------------|-----------|-------|
| 1,000 | 10,000回 | $1.50 + $1.20 = $2.70 | **約¥405** |
| 5,000 | 50,000回 | $7.50 + $6.00 = $13.50 | **約¥2,025** |
| 20,000 | 200,000回 | $30.00 + $24.00 = $54.00 | **約¥8,100** |

> ※ マッチング説明文をキャッシュ（Redis）すれば費用を50〜80%削減可能。

#### 特徴

- **強み**: 「なぜマッチしたか」を自然言語で説明できる。ユーザー体験の向上に直結する
- **弱み**: レイテンシが500ms〜2秒以上かかる。オンライン推論はUX劣化リスクがある。コストが他のアプローチより高い
- **導入コスト**: 低（APIキー設定のみ）。ただし品質担保のためプロンプトエンジニアリングに時間を要する

---

### 2.3 協調フィルタリング（SageMaker / Lambda）

#### 概要

「同じ曲をやりたいユーザーAとBは、Aが評価したマッチング相手をBも気に入る可能性が高い」という行動データに基づく推薦手法です。

| 項目 | 内容 |
|------|------|
| **インフラ** | SageMaker（学習・推論）または Lambda + DynamoDB（軽量実装） |
| **学習データ** | マッチングのスワイプ行動（いいね/スキップ）、チャット開始率、セッション継続率 |
| **学習頻度** | 週1回バッチ学習（初期） |
| **最小有効データ量** | ユーザーあたり最低20〜50件の行動データが必要 |

#### 費用試算（SageMaker ml.m5.large使用の場合）

| MAU | 学習時間（月） | 推論コスト | 合計月額（ドル） | 円換算 |
|-----|--------------|---------|----------------|-------|
| 1,000 | 2時間 × $0.115 | $5.00 | $5.23 | **約¥785** |
| 5,000 | 4時間 × $0.115 | $15.00 | $15.46 | **約¥2,320** |
| 20,000 | 8時間 × $0.115 | $50.00 | $50.92 | **約¥7,640** |

> Lambda + DynamoDB による軽量実装（Matrix Factorizationを自前実装）の場合、MAU 5,000規模でも月額¥500以下に抑えられますが、精度は低下します。

#### 特徴

- **強み**: 明示的なプロフィール入力に依存せず、行動から学習する。コールドスタート後は精度が急上昇する
- **弱み**: データが少ない初期（MAU 1,000未満）は機能しない。「行動の少ないユーザー」には依然コールドスタート問題が残る
- **導入コスト**: 高（SageMaker設定、データパイプライン構築、モデル評価基盤が必要）

---

### 2.4 ハイブリッド（現行スコアリング + Embeddingブースト）

#### 概要

現行の6要素スコアリングをベースとして維持しつつ、Embeddingの類似度スコアを加点ブーストとして追加するアプローチです。

```
最終スコア = 現行スコア(100点換算) × 0.85 + Embedding類似度スコア × 0.15
```

| 項目 | 内容 |
|------|------|
| **ベース** | 現行6要素スコアリング（変更なし） |
| **追加要素** | Embeddingコサイン類似度を最大15ptのブーストとして加算 |
| **対象** | 自己紹介文・ジャンルテキスト・アーティスト名の埋め込み |
| **特徴** | 現行の説明可能性・低コストを維持しつつ、テキスト意味理解を付加できる |

#### 費用試算

2.1のEmbeddingと同等（月額¥3〜¥60）。現行インフラへの変更が最小限で済む。

#### 特徴

- **強み**: リスク最小の導入経路。現行スコアの説明可能性を損なわない。コストが最も低い
- **弱み**: ブーストの重みの最適化が必要。Embeddingのみのアプローチより改善幅が限定的
- **導入コスト**: 低〜中（Embedding生成バッチ + pgvector + スコア統合ロジックの改修）

## 4. 段階的導入ロードマップ

スタートアップとして、リスク最小・学習優先の段階的導入を推奨します。

### 4.0 方針変更: ローンチからAI導入

**後からAIを追加する場合のコスト（エンジニア工数）:**

| 作業 | 工数見積もり |
|------|------------|
| DBスキーマ変更（embeddingカラム追加・マイグレーション） | 1〜2日 |
| pgvector導入・RDSアップグレード + 既存データへの一括Embedding生成 | 2〜3日 |
| マッチングロジック書き換え（非AI → ハイブリッド） | 2〜3日 |
| A/Bテスト基盤構築（効果測定のための実験フレームワーク） | 3〜5日 |
| **合計** | **エンジニア2〜4週間** |

**最初からハイブリッドを組み込む場合の追加コスト（インフラ）:**

| 項目 | 月額 |
|------|------|
| RDS db.t4g.smallへのアップグレード（pgvector対応） | +¥3,000〜5,000 |
| OpenAI Embedding API（MAU 2,000規模） | ¥3〜10 |
| **合計追加インフラコスト** | **¥3,003〜5,010/月** |

**結論:** 後からの移行工数（エンジニア2〜4週間 = 実質コスト数十万円）は、最初からの追加インフラコスト（¥3,000〜5,000/月）を大きく上回ります。さらに、ローンチ初日からEmbeddingデータが蓄積されるため、MAU成長とともに精度向上の恩恵を早期に受けられます。

---

### Phase 1: ローンチ（MAU 0〜2,000）

**方針: ハイブリッド（現行スコアリング + Embedding）をローンチから採用。**

| 項目 | 内容 |
|------|------|
| アルゴリズム | 現行スコアリング × 0.85 + Embedding類似度 × 0.15 |
| 月額費用 | ¥5,000〜7,000（RDSアップグレード含む） |
| 優先タスク | ユーザーの行動データ収集基盤の構築 |
| 判断基準 | MAU 2,000到達・スワイプ行動データ蓄積開始 |

**この段階でやること:**
- マッチングに関するユーザーフィードバックの収集（いいね/スキップ率、チャット開始率）
- プロフィール入力率の改善（曲未登録ユーザーの曲スコア問題への対応としてオンボーディング改善を優先）
- ローンチ時から全ユーザーのEmbeddingを生成・蓄積（データ資産の早期形成）

---

### Phase 2: 成長期（MAU 2,000〜10,000）

**方針: LLM推薦テキストをPro会員向けに追加。**

| 項目 | 内容 |
|------|------|
| アルゴリズム | Phase 1と同様（現行スコアリング × 0.85 + Embedding × 0.15）。Embedding重みの最適化 |
| 追加月額費用 | +¥1,000〜2,000（LLM推薦テキストのAPI費用） |
| 想定精度向上 | Phase 1比 +3〜5%（Embedding重み最適化 + LLM推薦によるUX向上） |
| 判断基準 | MAU 10,000到達・協調フィルタリング用データ十分化 |

**追加実装ステップ:**

1. A/Bテストでembedding重み（0.15）の最適化（チャット開始率・マッチング成立率を比較）
2. マッチング画面のトップカードに「なぜマッチしたか」説明文を夜間バッチで生成しRedisキャッシュ
3. LLM推薦対象をPro会員のみにすることで費用を抑制（MAU 5,000・Pro率10%の場合、500ユーザー × 月10回 = 約¥1,000/月）

---

### Phase 3: スケール期（MAU 10,000以上）

**方針: 協調フィルタリングの追加によるフルハイブリッド化。**

| 項目 | 内容 |
|------|------|
| アルゴリズム | 現行スコアリング × 0.60 + Embedding × 0.20 + 協調フィルタリング × 0.20 |
| 追加月額費用 | ¥7,000〜15,000（SageMaker または Lambda実装に依存） |
| 想定精度向上 | +20〜30%（Phase 1比） |
| 前提条件 | ユーザーあたり平均20件以上の行動データ蓄積 |

**実装ステップ:**

1. スワイプ行動ログのデータパイプライン整備（既存RDSから分析用データ抽出）
2. Matrix Factorizationモデルの実装（Lambda + S3でシンプルに開始）
3. 週次バッチ学習 → 推薦スコアをRedisにプリキャッシュ
4. A/Bテストで協調フィルタリングの有効性を検証
5. 効果確認後にSageMakerへ移行し精度向上

### 5.5 意思決定フレームワーク

**基本方針: ローンチからハイブリッドAIを採用する（後からの移行工数 > 最初からの追加インフラコスト）**

```

    → 行動データが十分蓄積されていることを確認（平均20件以上/ユーザー）

```
