# Groove Match - 監視・障害検知設計

> 本ドキュメントは、Groove Matchの障害をいかに早く検知し、影響を最小限に抑えるかを設計します。
> 基本的な監視閾値は [運用・保守](./02_運用・保守.md) を参照してください。

---

## 2. 外形監視（Synthetic Monitoring）

ユーザーと同じ視点でサービスの正常性を確認する**能動的な監視**です。サーバーのメトリクスが正常でもユーザーから見えない障害（DNS障害、CDN障害、SSL証明書期限切れ等）を検知できます。

### 2.1 CloudWatch Synthetics Canary

| Canary名 | チェック内容 | 実行頻度 | 成功条件 | 障害時 |
|---------|------------|---------|---------|--------|
| **health-check** | `GET /api/health` → 200応答 | 1分間隔 | レスポンス200 + 応答時間 < 3秒 | Slack + メール即時通知 |
| **login-flow** | ランディングページ → ログイン画面表示 | 5分間隔 | ページ描画完了 + 主要要素の存在確認 | Slack通知 |
| **api-matching** | `GET /api/matching` → 正常レスポンス | 5分間隔 | レスポンス200 + JSON構造検証 | Slack通知 |
| **ssl-check** | SSL証明書の有効期限確認 | 1日1回 | 有効期限が30日以上先 | メール通知（14日前から毎日） |

### 2.2 ヘルスチェックエンドポイント

アプリケーション内に専用のヘルスチェックAPIを実装する。

```
GET /api/health

レスポンス（正常時）:
{
  "status": "healthy",
  "timestamp": "2026-03-20T12:00:00Z",
  "checks": {
    "database": { "status": "up", "latency_ms": 5 },
    "redis": { "status": "up", "latency_ms": 2 },
    "spotify_api": { "status": "up", "latency_ms": 120 },
    "stripe_api": { "status": "up", "latency_ms": 80 }
  }
}

レスポンス（一部障害時）:
{
  "status": "degraded",
  "timestamp": "2026-03-20T12:00:00Z",
  "checks": {
    "database": { "status": "up", "latency_ms": 5 },
    "redis": { "status": "down", "error": "Connection refused" },
    "spotify_api": { "status": "up", "latency_ms": 120 },
    "stripe_api": { "status": "up", "latency_ms": 80 }
  }
}
```

| ステータス | 条件 | HTTPステータス |
|-----------|------|:------------:|
| `healthy` | 全チェック正常 | 200 |
| `degraded` | 一部チェック失敗（サービスは継続可能） | 200 |
| `unhealthy` | 必須チェック失敗（DB等） | 503 |

> ALBのヘルスチェックはこのエンドポイントを使用する。`unhealthy` の場合、ALBはそのタスクをローテーションから除外する。

## 4. アプリケーション監視

### 4.1 構造化ログ

全アプリケーションログをJSON形式で出力し、CloudWatch Logsに集約する。

```json
{
  "timestamp": "2026-03-20T12:00:00.000Z",
  "level": "ERROR",
  "service": "groove-match-api",
  "method": "POST",
  "path": "/api/matching/request",
  "status": 500,
  "duration_ms": 1250,
  "user_id": "usr_abc123",
  "error": "DatabaseConnectionError",
  "message": "Connection to RDS timed out after 5000ms",
  "request_id": "req_xyz789",
  "trace_id": "trace_abc"
}
```

### 4.2 ログベースの障害検知（CloudWatch Logs Metric Filter）

CloudWatch Logsからエラーパターンを自動検出し、メトリクス化する。

| フィルターパターン | メトリクス名 | 閾値 | アクション |
|-----------------|------------|------|-----------|
| `{ $.level = "ERROR" }` | `AppErrorCount` | > 10件/5分 | Slack通知 |
| `{ $.level = "ERROR" }` | `AppErrorCount` | > 50件/5分 | Slack + メール通知（Critical） |
| `{ $.error = "DatabaseConnectionError" }` | `DBConnectionErrors` | > 0件/1分 | Slack通知（即時） |
| `{ $.status = 503 }` | `ServiceUnavailable` | > 0件/1分 | Slack + メール通知（Critical） |
| `{ $.path = "/api/stripe/webhook" && $.status >= 400 }` | `StripeWebhookErrors` | > 0件/5分 | Slack通知（決済障害の可能性） |
| `{ $.duration_ms > 5000 }` | `SlowRequests` | > 10件/5分 | Slack通知（性能劣化） |
| `"OutOfMemoryError"` | `OOMErrors` | > 0件 | Slack + メール通知（Critical） |

### 4.3 エラートラッキング（Sentry）

フロントエンドとバックエンドのエラーをSentryで自動収集する。

| 設定 | 内容 |
|------|------|
| 対象 | JavaScript/TypeScriptの未ハンドルエラー、Reactコンポーネントエラー |
| 環境 | production、staging（devは除外） |
| アラート | 新しいエラーが発生した時にSlack通知 |
| サンプリング | 本番: 100%（全エラー収集）、パフォーマンストレース: 10% |
| PII除去 | ユーザーのメールアドレス・パスワードをエラーレポートから除去 |

## 5. ビジネスメトリクス監視

技術的には正常でも、ビジネス指標の異常な変動は障害の兆候である場合がある。

### 5.1 日次チェック（CloudWatch カスタムメトリクス）

| メトリクス | 算出方法 | 異常条件 | アクション |
|-----------|---------|---------|-----------|
| 新規登録数 | 当日の登録完了数 | 前週同曜日比 **50%以下** | Slack通知（登録障害の可能性） |
| ログイン成功率 | ログイン成功 / ログイン試行 | **90%以下** | Slack通知（認証障害の可能性） |
| マッチング実行数 | 当日のマッチング検索数 | 前週同曜日比 **50%以下** | Slack通知 |
| 決済成功率 | Stripe Webhook成功 / 決済試行 | **95%以下** | Slack + メール通知（決済障害） |
| チャット送信数 | 当日のメッセージ送信数 | 前週同曜日比 **50%以下** | Slack通知（WebSocket障害の可能性） |

### 5.2 実装方法

```
毎日 9:00 JST に EventBridge → Lambda が実行
    ↓
Lambda が RDS から前日のメトリクスを集計
    ↓
CloudWatch にカスタムメトリクスとして PUT
    ↓
CloudWatch Alarms で異常を検知
    ↓
SNS → Slack 通知
```

## 6. セキュリティ監視

### 6.1 AWS CloudTrail

全アカウントでCloudTrailを有効化し、APIコールを記録する。

| 検知対象 | 検知方法 | アクション |
|---------|---------|-----------|
| IAMポリシー変更 | CloudTrail + EventBridge | Slack + メール通知（即時） |
| セキュリティグループ変更 | CloudTrail + EventBridge | Slack + メール通知（即時） |
| RDS/S3のパブリックアクセス変更 | AWS Config Rules | Slack + メール通知（Critical） |
| rootユーザーログイン | CloudTrail + EventBridge | Slack + メール通知（Critical） |
| 不正なAPI呼び出し（AccessDenied多発） | CloudTrail Insights | Slack通知 |

### 6.2 WAFログ

| 検知対象 | 検知方法 | アクション |
|---------|---------|-----------|
| SQLインジェクション試行 | WAF ルール（AWSManagedRulesSQLiRuleSet） | ブロック + ログ記録 |
| XSS試行 | WAF ルール（AWSManagedRulesCommonRuleSet） | ブロック + ログ記録 |
| レートリミット超過 | WAF レートベースルール（1,000 req/5分/IP） | 一時ブロック + Slack通知 |
| ボット検出 | WAF ボットコントロール | ブロック + ログ記録 |

### 6.3 アプリケーションセキュリティ

| 検知対象 | 検知方法 | アクション |
|---------|---------|-----------|
| 連続ログイン失敗 | Cognito + カスタムメトリクス（同一IP/メールで5回失敗） | アカウント一時ロック（30分） + ログ |
| 異常な大量リクエスト | ALBアクセスログ分析 | WAFでIP一時ブロック |
| 不審なWebhook呼び出し | Stripe署名検証失敗 | ログ記録 + 3回連続で Slack通知 |

## 7. 通知・エスカレーション設計

### 7.1 通知先の分類

| 重要度 | 通知チャネル | 応答期待時間 | 例 |
|--------|------------|------------|-----|
| **Critical** | Slack + メール + **LINE通知** | **30分以内** | サービス全停止、DB障害、全タスク停止 |
| **High** | Slack + メール + **LINE通知** | **1時間以内** | 5xxエラー率5%超、決済障害 |
| **Medium** | Slack | **4時間以内** | CPU高負荷、キャッシュヒット率低下 |
| **Low** | Slack（日次サマリー） | **翌営業日** | ストレージ警告、パフォーマンス劣化傾向 |

### 7.1.1 LINE通知（Critical/High 障害のみ）

重大障害（Critical・High）が発生した場合、開発者の個人LINEにも即時通知する。PCを開いていない状況でもスマートフォンで障害を検知できる。

**方式: LINE Notify API + AWS Lambda**

```
CloudWatch Alarm (Critical/High)
    ↓
SNS トピック (critical-alerts)
    ↓
Lambda (LINE通知ハンドラー)
    ↓
LINE Notify API (https://notify-api.line.me/api/notify)
    ↓
開発者のLINEに通知メッセージ
```

**LINE Notify の設定手順:**

1. [LINE Notify](https://notify-bot.line.me/) にログイン
2. 「トークンを発行する」でパーソナルアクセストークンを生成
3. トークンを AWS Secrets Manager に保存（`line-notify-token`）
4. Lambda関数を作成:

```python
# Lambda: line-notify-critical-alerts
import json
import urllib.request
import boto3

def lambda_handler(event, context):
    # Secrets Manager からLINE Notifyトークンを取得
    client = boto3.client('secretsmanager')
    token = client.get_secret_value(SecretId='line-notify-token')['SecretString']

    # SNSメッセージからアラート情報を抽出
    message = json.loads(event['Records'][0]['Sns']['Message'])
    alarm_name = message.get('AlarmName', 'Unknown')
    reason = message.get('NewStateReason', 'No details')

    # LINE Notifyに送信
    text = f"\n🚨 Groove Match 障害検知\n\nアラーム: {alarm_name}\n詳細: {reason}\n\nAWS Console で確認してください"

    req = urllib.request.Request(
        'https://notify-api.line.me/api/notify',
        data=f'message={text}'.encode('utf-8'),
        headers={'Authorization': f'Bearer {token}'},
        method='POST'
    )
    urllib.request.urlopen(req)
```

5. SNSトピック `critical-alerts` のサブスクリプションにLambdaを追加

**LINE通知の対象アラーム（Critical/Highのみ）:**

| アラーム | 重要度 | LINE通知メッセージ例 |
|---------|:------:|------------------|
| ECS全タスク停止 | Critical | 「ECSタスクが全て停止しました。サービスがダウンしています」 |
| RDS接続障害 | Critical | 「RDSへの接続に失敗しています。DB障害の可能性」 |
| 5xxエラー率 > 5% | Critical | 「サーバーエラー率が5%を超えました」 |
| ヘルスチェック失敗 | Critical | 「ヘルスチェックが連続失敗。サービス異常」 |
| 決済障害 | High | 「Stripe Webhookでエラーが発生。決済に影響の可能性」 |
| OOMエラー | High | 「メモリ不足エラーが発生しました」 |

> **LINE Notify は2025年3月末でサービス終了予定**のため、代替として **LINE Messaging API**（公式アカウント経由）への移行を検討する。Messaging APIは無料枠（月200通）で十分対応可能。

### 7.2 Slack通知チャネル設計

| チャネル | 用途 | 投稿されるアラート |
|---------|------|------------------|
| `#alerts-critical` | 即時対応が必要な障害 | サービス停止、DB障害、決済障害 |
| `#alerts-warning` | 注意が必要な状態 | CPU高負荷、エラー率増加、ストレージ警告 |
| `#alerts-security` | セキュリティイベント | IAM変更、WAFブロック、不正ログイン試行 |
| `#alerts-business` | ビジネスメトリクス異常 | 登録数激減、決済成功率低下 |
| `#deploy-notifications` | デプロイ通知 | デプロイ開始/完了/失敗 |

### 7.3 アラート疲れの防止

| 対策 | 説明 |
|------|------|
| **重複抑制** | 同一アラームの再通知は最短15分間隔 |
| **自動回復通知** | アラーム状態が解消されたら「回復しました」通知を送信 |
| **営業時間外のLow抑制** | Low重要度は22時〜8時の間は通知しない（翌朝サマリーにまとめる） |
| **日次サマリー** | 毎朝9時に前日のアラート件数・種別・解決状況を `#alerts-warning` に投稿 |

## 8. 障害対応ランブック

### 8.1 共通手順

```
1. アラート受信
    ↓
2. 影響範囲の確認
   - ユーザーへの影響はあるか？
   - どの機能が影響を受けているか？
   - いつから発生しているか？
    ↓
3. 障害レベルの判定（Critical/High/Medium/Low）
    ↓
4. 対応実施
    ↓
5. 復旧確認（ヘルスチェックエンドポイント + 手動確認）
    ↓
6. ポストモーテム作成（Criticalの場合は必須）
```

### 8.2 頻出障害と対応手順

#### ECSタスク停止

| 手順 | コマンド / アクション |
|------|---------------------|
| 1. タスク停止理由を確認 | `aws ecs describe-tasks --tasks <task-arn>` → `stoppedReason` |
| 2. OOMの場合 | タスク定義のメモリを増量してデプロイ |
| 3. ヘルスチェック失敗の場合 | `/api/health` を手動確認。DB/Redis接続を確認 |
| 4. イメージPull失敗の場合 | ECRのイメージ存在確認。ECRエンドポイントの疎通確認 |

#### RDS接続障害

| 手順 | コマンド / アクション |
|------|---------------------|
| 1. RDSステータス確認 | `aws rds describe-db-instances` → Status |
| 2. コネクション数確認 | CloudWatchメトリクス `DatabaseConnections` |
| 3. 接続上限の場合 | RDS Proxyの接続プーリング設定を確認。不要な接続をkill |
| 4. RDS停止の場合 | 自動再起動を待つ。手動再起動: `aws rds reboot-db-instance` |
| 5. データ復旧が必要な場合 | 最新スナップショットからリストア |

#### 決済障害（Stripe Webhook失敗）

| 手順 | コマンド / アクション |
|------|---------------------|
| 1. Stripeダッシュボードでイベントを確認 | Stripe Dashboard → Developers → Events |
| 2. Webhook署名エラーの場合 | STRIPE_WEBHOOK_SECRET がSecrets Managerで正しいか確認 |
| 3. エンドポイント到達不可の場合 | ALB/ECSの稼働を確認。セキュリティグループでStripe IPを許可 |
| 4. Stripe側障害の場合 | [status.stripe.com](https://status.stripe.com/) を確認。復旧を待つ |
| 5. 失敗イベントの再送 | Stripeダッシュボードから手動リトライ |

#### CDN障害（CloudFront 5xxエラー）

| 手順 | コマンド / アクション |
|------|---------------------|
| 1. CloudFrontエラー率を確認 | CloudWatchメトリクス `5xxErrorRate` |
| 2. オリジン（ALB）の稼働確認 | ALBのヘルシーホスト数を確認 |
| 3. キャッシュ起因の場合 | CloudFrontのキャッシュを無効化（Invalidation） |
| 4. SSL証明書期限切れの場合 | ACMで証明書を更新 → CloudFrontに適用 |

#### WebSocket障害（チャット停止）

| 手順 | コマンド / アクション |
|------|---------------------|
| 1. API Gateway WebSocketのステータス確認 | AWS Console → API Gateway → WebSocket API |
| 2. Lambda実行エラーの確認 | CloudWatch Logs で Lambda エラーログ確認 |
| 3. DynamoDB/RDS接続の確認 | チャット関連テーブルへのクエリを手動実行 |
| 4. 一時対応 | API Gatewayのステージを再デプロイ |

### 8.3 ポストモーテムテンプレート

Critical障害の発生後、72時間以内に作成する。

