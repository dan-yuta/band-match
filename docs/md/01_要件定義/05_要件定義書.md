## 1. プロジェクト概要・ビジョン・KPI

### 1.1 プロジェクト概要

Groove Match は **コピーバンドメンバーマッチング & ライブ出演促進プラットフォーム** である。アマチュアミュージシャンが「やりたい曲・やりたいバンドのジャンル」を軸にメンバーを見つけ、バンドを結成し、ライブ出演まで完結できるコミュニティアプリを提供する。

| 項目 | 内容 |
|------|------|
| サービス名 | Groove Match |
| ドメイン | groovematch.jp |
| デモURL | https://groove-match.netlify.app/ |
| プラットフォーム | Web / iOS / Android（クロスプラットフォーム） |
| フロントエンド | Next.js (App Router) + TypeScript + Tailwind CSS v4 |
| ネイティブ対応 | Capacitor |
| バックエンド | Next.js API Routes / Server Actions + Prisma |
| データベース | RDS PostgreSQL |
| 認証 | AWS Cognito |
| インフラ | AWS（ECS Fargate, CloudFront, RDS, ElastiCache, API Gateway WebSocket） |

関連ドキュメント: [アプリ概要](./01_アプリ概要.md) | [機能一覧](./02_機能一覧.md)

---

### 1.2 ビジョン・ミッション

> **「やりたい曲で仲間を見つけて、コピバン組んで、ライブに出よう」**

| 区分 | 内容 |
|------|------|
| ビジョン | 「好きな音楽で繋がる」コミュニティを全国に広げ、アマチュアバンドシーンを活性化する |
| ミッション | バンドメンバー探しの非効率（SNS検索、掲示板投稿の手間）をなくし、マッチングから本番ライブまでを一気通貫でサポートする |
| ポジショニング | ジャムバンド・コピーバンド特化型（既存の汎用ミュージシャンマッチングとの差別化） |

## 4. マッチング要件

参考: [マッチングアルゴリズム詳細](../02_基本設計/01_マッチングアルゴリズム.md) | [AI対非AIコスト比較](../02_基本設計/04_AI対非AIコスト比較.md)

### 4.1 Version A：ハイブリッドスコアリング方式（MVPローンチから採用）

MVP（最初のリリース）から、ルールベーススコアリングにEmbeddingブーストを組み合わせたハイブリッド方式を採用する。後からAIを追加すると移行工数（DBスキーマ変更・ロジック書き換え・A/Bテスト基盤構築でエンジニア2〜4週間）が発生するため、最初からハイブリッドを組み込む方がトータルコストで有利である。

#### スコアリング要素

| 要素 | 最大スコア | 重み付け理由 |
|------|-----------|-------------|
| 地域マッチ | 40点 | 同一地域でなければスタジオ練習が不可能。最重要 |
| 楽器マッチ | 20点 | バンドに必要な楽器の補完性を評価 |
| ジャンルマッチ | 20点 | 音楽的方向性の一致度 |
| スキルレベル | 15点 | 近似レベルで練習しやすさが向上 |
| スケジュールマッチ | 30点 | 活動可能時間の重複度 |
| やりたい曲マッチ | 35点 | コピバン特有の最重要要素。曲が合わないとバンドが成立しない |
| **合計（生スコア）** | **160点** | |
| **表示スコア** | **100点** | 生スコアを100点満点にクランプ（正規化） |

#### スコア計算式

```
rawScore = 地域スコア + 楽器スコア + ジャンルスコア + スキルスコア + スケジュールスコア + 曲スコア
displayScore = Math.min(100, Math.round((rawScore / 160) * 100))
```

#### 地域スコア詳細

| 条件 | スコア |
|------|--------|
| 同一市区町村 | 40点 |
| 同一都道府県 | 25点 |
| 隣接都道府県 | 10点 |
| それ以外 | 0点 |

#### やりたい曲スコア詳細（35点）

| 条件 | スコア |
|------|--------|
| 完全一致する曲が3曲以上 | 35点 |
| 完全一致する曲が2曲 | 25点 |
| 完全一致する曲が1曲 | 15点 |
| アーティスト一致（曲は不一致） | 5点 |
| ジャンルのみ一致 | 2点 |
| 不一致 | 0点 |

### 4.2 Version B：LLM推薦テキスト強化（Phase 2 以降）

Phase 2 以降、ハイブリッドスコアリング（Version A）をベースに、LLM（GPT-4o-mini）によるマッチング理由の自然言語説明文をPro会員向けに追加する。EmbeddingによるAIマッチング自体はローンチ（Version A）から組み込み済みのため、Phase 2では精度向上ではなくUX向上（説明文生成）が主目的となる。

| 項目 | 内容 |
|------|------|
| 使用モデル（Embedding） | `text-embedding-3-small`（OpenAI）※ローンチから適用済み |

| **Embedding** | エンベディング | テキストをベクトル（数値列）に変換する技術。ローンチからハイブリッドマッチングに使用（OpenAI text-embedding-3-small） |
| **pgvector** | ピージーベクター | PostgreSQLにベクトル検索機能を追加する拡張機能。Embedding類似度計算に使用（ローンチから導入） |
| **CI/CD** | シーアイシーディー | Continuous Integration / Continuous Delivery（継続的インテグレーション/継続的デリバリー）。コード変更を自動テスト・デプロイする仕組み（GitHub Actions） |
| **IaC** | アイエーシー | Infrastructure as Code（コードとしてのインフラ）。インフラ構成をコードで定義・管理する手法（CloudFormation） |
| **CloudFormation** | クラウドフォーメーション | AWSのIaCサービス。YAMLまたはJSONでAWSリソースを定義・デプロイできる |
| **Pro会員** | プロかいいん | 本アプリの有料プランユーザー。月額 ¥980 で無制限マッチング・チャット等の拡張機能を利用可能 |
| **コピーバンド** | ─ | 既存アーティストの楽曲を演奏するバンド。略称「コピバン」。本サービスの主要ターゲット |
| **レピュテーションスコア** | ─ | ユーザーの信頼度を示す内部スコア（0〜100）。通報・違反で低下し、問題なし利用で自然回復 |
| **DAG** | ディーエージー | Directed Acyclic Graph（有向非巡回グラフ）。タスクの依存関係を表すデータ構造。ワークフロー管理等で使用 |
| **SLA** | エスエルエー | Service Level Agreement（サービスレベル合意）。対応時間・可用性等のサービス品質を定めた合意事項 |
| **RTO** | アールティーオー | Recovery Time Objective（目標復旧時間）。障害発生からサービス復旧までの目標時間 |
| **RPO** | アールピーオー | Recovery Point Objective（目標復旧時点）。障害発生時に許容できる最大データ損失時間 |
