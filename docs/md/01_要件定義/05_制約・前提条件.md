## 6. ローンチ戦略: クローズドベータ（招待制テスト）

### 方針

ローンチ後の**最初の1ヶ月間はクローズドベータ**とし、**招待制で最大100名のテストユーザー**に限定して提供する。

| 項目 | 内容 |
|------|------|
| 期間 | ローンチ後1ヶ月間 |
| ユーザー数上限 | **100名**（招待コードによる登録のみ受付） |
| 対象 | 音楽関連コミュニティ・軽音サークル・SNS募集で選定 |
| プラン | **全員Proプラン無料提供**（テスト期間中） |
| 目的 | 致命的バグの発見、UX検証、マッチング精度の評価 |

### 招待制の実装

| 実装 | 説明 |
|------|------|
| 登録時に招待コード必須 | 招待コードがなければ登録不可（通常の新規登録フォームは非表示） |
| 管理者が100件の招待コードを事前発行 | `/admin` ダッシュボードから一括生成 |
| 使用済みコードの追跡 | 誰がいつ使ったかを記録。テストユーザーの属性分析に活用 |
| ベータ終了後 | 招待制を解除し、一般公開に移行 |

### ユーザーテスト観点

テストユーザーには以下の観点でフィードバックを収集する。

#### 機能テスト

| テスト項目 | 確認観点 | 収集方法 |
|-----------|---------|---------|
| ユーザー登録フロー | 5ステップの登録は長すぎないか、離脱ポイントはどこか | 登録完了率、ステップ別離脱率 |
| マッチング精度 | 提案されたユーザーは「一緒にやりたい」と思えるか | マッチング後のいいね率、スキップ率 |
| チャット体験 | 練習日程の調整はスムーズか、機能は十分か | チャット開始率、メッセージ数 |
| コピバン結成 | セットリスト管理・楽器スロットは使いやすいか | バンド作成数、メンバー充足率 |
| イベントボード | 情報は十分か、応募しやすいか | イベント閲覧数、応募率 |
| 練習トラッカー | ストリーク・バッジは継続モチベーションになるか | 練習記録の継続率 |

#### UX・デザインテスト

| テスト項目 | 確認観点 |
|-----------|---------|
| 初回体験（FTUE） | 初めて使った時に何をすれば良いか分かるか |
| ナビゲーション | 目的の画面にたどり着けるか |
| モバイル操作性 | スマートフォンでの操作は快適か |
| 表示速度 | 画面の読み込みは体感として遅くないか |
| ダークモードUI | 文字の視認性、コントラストは十分か |

#### セキュリティ・安全テスト

| テスト項目 | 確認観点 |
|-----------|---------|
| 通報フロー | 通報ボタンの発見しやすさ、操作の直感性 |
| ブロック機能 | ブロック後にマッチング・タイムラインから完全に消えるか |
| 禁止ワードフィルタ | 勧誘・個人情報のパターンが正しく検出されるか |
| 安全アドバイス | 初回対面時のアドバイスは適切なタイミングで表示されるか |

#### フィードバック収集方法

| 方法 | タイミング | 内容 |
|------|-----------|------|
| **アプリ内アンケート** | 登録後1週間、2週間、4週間 | NPS（推奨度）、機能満足度、改善要望 |
| **1on1インタビュー** | ベータ期間中に10名程度 | 深掘りヒアリング（操作を観察しながら） |
| **バグ報告フォーム** | 常時 | アプリ内からバグ報告できる導線 |
| **利用データ分析** | ベータ終了時 | 登録完了率、DAU推移、機能別利用率、離脱ポイント |

### クローズドベータ → 一般公開の判定基準

| 指標 | 一般公開の条件 |
|------|-------------|
| 致命的バグ | **0件**（クリティカルなバグが全て修正済み） |
| 登録完了率 | **70%以上**（登録フォーム開始→完了） |
| DAU/MAU比 | **20%以上**（テストユーザーの継続利用） |
| マッチング成立率 | **30%以上**（いいね送信→相互マッチング） |
| NPS | **0以上**（推奨者 > 批判者） |
| セキュリティ | 通報・ブロック機能が正常動作。情報漏洩ゼロ |
