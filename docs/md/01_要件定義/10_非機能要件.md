# Groove Match - 非機能要件

> 本ドキュメントは「システムに求める品質」を定義します。
> 各要件の実現方法（HOW）は [インフラ構成](../02_基本設計/02_インフラ構成.md) を参照してください。

---

## 1. 性能要件

| 項目 | 要件 | 測定方法 |
|------|------|---------|
| ページ読み込み時間 | **P95 < 2秒** | CloudFront + LCP基準 |
| API応答時間 | **P95 < 500ms** | CloudWatch メトリクス |
| マッチングスコア計算 | **< 200ms**（100ユーザー分） | サーバーサイド計測 |
| WebSocketメッセージ配信 | **< 100ms** | API Gateway メトリクス |

---

## 2. 可用性・信頼性要件

| 項目 | 要件 |
|------|------|
| 稼働率 | **99.9%**（月間ダウンタイム約43分以内） |
| 計画メンテナンス | 深夜2:00-5:00（JST）、ユーザーへ24時間前に事前通知 |
| 障害復旧目標（RTO） | **1時間以内** |
| 障害復旧目標（RPO） | **24時間以内** |
| ヘルスチェック | 30秒間隔 |

---

## 3. スケーラビリティ要件

| フェーズ | MAU | 同時接続 |
|---------|-----|---------|
| 初期 | 〜1,000 | 〜100 |
| 成長期 | 1,000〜5,000 | 〜1,000 |
| 拡大期 | 5,000〜20,000 | 〜3,000 |

RDS → Aurora への移行トリガー: **MAU 5,000達成時または書き込みQPS 100超過時**

---

## 4. データ保全要件

| 項目 | 要件 |
|------|------|
| DB自動バックアップ | **7日間保持**（日次） |
| 手動スナップショット | **週1回**（30日保持） |
| クロスリージョンバックアップ | 東京 → バージニアへ複製 |
| バックアップ暗号化 | AES-256 |
| 復元テスト | **月1回** staging環境で実施 |

---

## 5. セキュリティ要件

| 項目 | 要件 |
|------|------|
| 認証方式 | AWS Cognito（JWT + リフレッシュトークン） |
| パスワード | 8文字以上、英数字混在必須 |
| 通信暗号化 | HTTPS必須（TLS 1.2以上） |
| データ暗号化（保存時） | AES-256（RDS、S3、バックアップ） |
| 決済データ | **カード情報はサーバーに一切保持しない**（PCI DSS SAQ-A） |
| API保護 | Rate Limiting、CORS、WAF |
| 脆弱性管理 | npm audit + Dependabot |

> 通報・ブロック・エスカレーション等のユーザー安全対策は [安全対策・リスク管理](./04_安全対策・リスク管理.md) を参照。
> PCI DSS準拠の詳細は [バックエンド設計](../03_詳細設計/02_バックエンド設計.md) を参照。

---

## 6. 保守性要件

| 項目 | 要件 |
|------|------|
| ログ保持 | 30日（CloudWatch Logs） |
| エラー通知 | リアルタイム（Slack + LINE） |
| デプロイ方式 | **ブルーグリーンデプロイ**（ダウンタイムゼロ） |
| IaC | CloudFormation（インフラ変更履歴をGitで管理） |
| CI/CD | GitHub Actions（テスト・ビルド・デプロイ自動化） |

---

## 7. 監視要件

| 項目 | 要件 |
|------|------|
| 外形監視 | 1分間隔でヘルスチェック |
| 障害検知 | CloudWatch Alarms + ログベース検知 |
| Critical障害通知 | Slack + メール + **LINE** |
| ビジネスメトリクス | 日次で登録数・ログイン成功率・決済成功率を自動チェック |
| ポストモーテム | Critical障害時に72時間以内に作成必須 |

> 監視の詳細設計は [監視・障害検知設計](../04_コスト・運用/03_監視・障害検知設計.md) を参照。
