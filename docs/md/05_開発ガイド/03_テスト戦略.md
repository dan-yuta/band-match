# Groove Match - テスト戦略

---

## 1. テスト方針

| 原則 | 内容 |
|------|------|
| **テストピラミッド** | ユニットテスト（多）> 統合テスト（中）> E2Eテスト（少）の比率を維持 |
| **段階的導入** | MVP段階では最低限のテストから開始し、機能追加に合わせてカバレッジを拡大 |
| **CI必須** | PR作成時にテストが自動実行され、失敗したPRはマージ不可 |
| **本番データ不使用** | テストには必ずモックデータ/テストデータを使用。本番DBに接続しない |

---

## 2. テストツール

| 種別 | ツール | 用途 |
|------|--------|------|
| ユニットテスト | **Vitest** | コンポーネント、ユーティリティ関数、マッチングアルゴリズム |
| コンポーネントテスト | **Testing Library** | Reactコンポーネントの描画・操作テスト |
| E2Eテスト | **Playwright** | ブラウザ上での統合テスト（登録、ログイン、マッチング等） |
| APIテスト | **Vitest + supertest** | API Routesの入出力検証 |
| 型チェック | **TypeScript (tsc)** | コンパイル時の型安全性チェック |
| リント | **ESLint** | コード品質・規約チェック |
| セキュリティ | **npm audit + Dependabot** | 依存パッケージの脆弱性検出 |

---

## 3. カバレッジ目標

| フェーズ | ユニットテスト | E2Eテスト | 対象 |
|---------|:------------:|:---------:|------|
| MVP（ローンチ時） | **60%以上** | 主要フロー5本 | 認証、マッチング、バンド作成 |
| フェーズ2（MAU 5,000） | **80%以上** | 主要フロー10本 | 決済、チャット、通知 |
| フェーズ3（MAU 20,000） | **90%以上** | 全フロー | 管理画面、エッジケース |

---

## 4. テスト対象と優先度

### 4.1 最優先（MVPリリース前に必須）

| テスト対象 | 種別 | テスト内容 |
|-----------|------|-----------|
| マッチングアルゴリズム | ユニット | 6要素スコア計算の正確性、クランプ処理、エッジケース（プロフィール未入力） |
| ユーザー登録 | E2E | 5ステップ登録の完走、バリデーション（メール形式、パスワード要件） |
| ログイン/ログアウト | E2E | 正常系、認証エラー、トークン期限切れ |
| Stripe Webhook | 統合 | 署名検証、checkout.session.completed、subscription.deleted |
| 認証ガード | ユニット | 未認証ユーザーのリダイレクト、管理者権限チェック |
| 入力バリデーション | ユニット | XSS対策、SQLインジェクション対策、パスワード要件 |

### 4.2 高優先（ローンチ後1ヶ月以内）

| テスト対象 | 種別 | テスト内容 |
|-----------|------|-----------|
| チャット送受信 | E2E | WebSocket接続、メッセージ送受信、既読更新 |
| バンド作成・管理 | E2E | 作成、メンバー招待、セットリスト管理 |
| イベント一覧・フィルター | E2E | 地域・日付・ジャンルフィルターの動作 |
| 通報・ブロック | 統合 | 通報フロー、ブロック後のマッチング除外 |
| プロフィール編集 | E2E | 楽器・ジャンル変更、画像アップロード |
| 禁止ワードフィルタ | ユニット | 勧誘キーワード検出、個人情報パターン検出 |

### 4.3 中優先

| テスト対象 | 種別 | テスト内容 |
|-----------|------|-----------|
| 練習トラッカー | E2E | 練習記録、ストリーク計算、バッジ獲得 |
| ランキング | ユニット | スコア集計、順位計算 |
| 通知 | 統合 | プッシュ通知送信、メール送信、頻度制限 |
| レピュテーションスコア | ユニット | スコア加減算、表示変換 |
| 招待コード | 統合 | コード生成、適用、トライアル延長 |

## 5. E2Eテストシナリオ（主要フロー）

### シナリオ1: 新規登録→マッチング→チャット

```
1. ランディングページを開く
2. 「無料で始める」をクリック
3. 5ステップの登録フォームを入力
4. 利用規約に同意
5. Stripe Checkout画面に遷移（テスト環境ではモック）
6. ダッシュボードに遷移することを確認
7. マッチング画面に遷移
8. マッチング候補が表示されることを確認
9. 「いいね」を送信
10. マッチング成立を確認（テストデータで相互いいね済みユーザーを用意）
11. チャット画面に遷移
12. メッセージを送信・表示を確認
```

### シナリオ2: コピバン結成→ライブ応募

```
1. ログイン
2. バンド一覧→「バンドを作成」
3. バンド情報を入力（名前、ジャンル、楽器スロット）
4. セットリストに曲を追加（Spotify検索モック）
5. バンド詳細画面で情報が正しく表示されることを確認
6. イベント一覧に遷移
7. フィルターで絞り込み
8. イベント詳細を開く
9. 「バンドで応募」を実行
```

### シナリオ3: 通報→ブロック

```
1. ログイン
2. マッチング画面でユーザー詳細を開く
3. 通報ボタンをクリック
4. カテゴリ選択、説明入力、送信
5. 通報完了メッセージを確認
6. ブロックボタンをクリック
7. ブロック確認ダイアログで確定
8. マッチング候補からブロックしたユーザーが消えていることを確認
```

## 6. CI/CDパイプラインでのテスト

```
PR作成 / Push
    │
    v
GitHub Actions
    │
    ├── (1) 型チェック: tsc --noEmit
    ├── (2) リント: eslint
    ├── (3) ユニットテスト: vitest run --coverage
    ├── (4) ビルド: next build
    └── (5) E2Eテスト: playwright test（staging環境）
         │
         ├── 全パス → マージ可能
         └── 失敗 → マージブロック + Slack通知
```

### テスト失敗時のルール

| 状況 | 対応 |
|------|------|
| ユニットテスト失敗 | PRをマージ不可。修正必須 |
| E2Eテスト失敗 | PRをマージ不可。環境起因の場合はリトライ1回 |
| カバレッジ低下 | 警告表示（マージはブロックしない。フェーズ2以降はブロック） |
| 型エラー | PRをマージ不可 |
| lint警告 | 警告表示（error はマージ不可） |

## 7. テスト環境

| 環境 | 用途 | データ |
|------|------|--------|
| ローカル | 開発中のテスト | モックデータ（localStorage） |
| CI（GitHub Actions） | PR時の自動テスト | テスト用フィクスチャ |
| staging | E2Eテスト、リリース前検証 | テスト用DB（本番データのコピーは禁止） |

### テストデータ方針

| データ | 方式 |
|--------|------|
| ユーザーデータ | `src/data/mockUsers.ts` のテストユーザー18名を使用 |
| バンドデータ | `src/data/mockBands.ts` のテストバンド7組 |
| Spotify API | テスト時はモックレスポンスを返す（実APIは呼ばない） |
| Stripe API | テスト環境のAPIキー（`sk_test_`）を使用 |
| DB | テスト開始前にシードデータを投入、テスト終了後にクリーンアップ |
