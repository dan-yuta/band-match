# Groove Match - モック→本番移行手順

> 現在のフロントエンドはモックデータ（localStorage）で動作しています。
> 本ドキュメントは、バックエンドAPI完成後にモック→本番接続へ移行する手順を定義します。

---

## 1. 移行の全体方針

| 方針 | 内容 |
|------|------|
| **段階的移行** | 全機能を一括で切り替えず、機能単位で段階的にAPI接続に移行 |
| **フィーチャーフラグ** | 環境変数 `NEXT_PUBLIC_USE_MOCK=true/false` でモック/API接続を切り替え可能にする |
| **並行稼働期間** | 各機能の移行中は、モック版とAPI版を両方テストできる状態を維持 |
| **ロールバック可能** | フィーチャーフラグを `true` に戻せば即座にモック版に戻せる |

---

## 2. フィーチャーフラグの実装

```typescript
// src/lib/config.ts
export const config = {
  useMock: process.env.NEXT_PUBLIC_USE_MOCK === 'true',
};

// 使用例: src/lib/api/users.ts
import { config } from '@/lib/config';
import { mockGetCurrentUser } from '@/data/mockUsers';

export async function getCurrentUser() {
  if (config.useMock) {
    return mockGetCurrentUser();  // 既存のモック関数
  }
  const res = await fetch('/api/users/me');
  return res.json();
}
```

| 環境 | `NEXT_PUBLIC_USE_MOCK` | 動作 |
|------|:---------------------:|------|
| 開発（初期） | `true` | モックデータで動作（現状と同じ） |
| 開発（API接続テスト中） | `false` | バックエンドAPIに接続 |
| staging | `false` | バックエンドAPIに接続 |
| prod | `false` | バックエンドAPIに接続 |

## 3. 移行順序（機能別）

バックエンド実装の優先度と依存関係を考慮し、以下の順序で移行する。

### Wave 1: 認証基盤（Week 5-6）

| 対象 | モック | API接続後 |
|------|--------|----------|
| ユーザー登録 | localStorage に保存 | Cognito User Pool に登録 |
| ログイン/ログアウト | localStorage チェック | Cognito JWT トークン発行 |
| 認証状態管理 | AuthContext（localStorage） | AuthContext（JWT + リフレッシュトークン） |

**移行手順:**
1. `src/lib/auth.tsx` にフィーチャーフラグ分岐を追加
2. Cognito接続版の `login()` / `register()` / `logout()` を実装
3. staging環境でテスト
4. フィーチャーフラグを `false` に切り替え

### Wave 2: プロフィール・マッチング（Week 9-12）

| 対象 | モック | API接続後 |
|------|--------|----------|
| プロフィール取得・更新 | localStorage の `groovematch-users` | `GET/PUT /api/users/me` |
| マッチング検索 | `src/lib/matching.ts` で計算 | `GET /api/matching` （サーバーサイド計算） |
| いいね送信 | localStorage に保存 | `POST /api/matching/request` |

**移行手順:**
1. `src/lib/storage.ts` の各関数にフィーチャーフラグ分岐を追加
2. API呼び出し版の関数を実装（`src/lib/api/` ディレクトリに配置）
3. Spotify API連携を実装（曲検索・メタ情報取得）
4. マッチングアルゴリズムをサーバーサイドに移植

### Wave 3: バンド・イベント・コミュニティ（Week 15-22）

| 対象 | モック | API接続後 |
|------|--------|----------|
| バンド一覧・作成 | `groovematch-bands` | `GET/POST /api/bands` |
| イベント一覧・作成 | `groovematch-events` | `GET/POST /api/events` |
| コミュニティ投稿 | `groovematch-posts` | `GET/POST /api/posts` |

### Wave 4: 決済・チャット・高度な機能（Week 19-30）

| 対象 | モック | API接続後 |
|------|--------|----------|
| サブスクリプション | モックUI | Stripe Checkout + Webhook |
| チャット | ─（未実装） | API Gateway WebSocket + Lambda |
| 練習トラッカー | localStorage | `POST /api/practice` |
| フレンド | localStorage | `POST /api/friends` |

## 4. データ移行

### 4.1 モックデータ → シードデータ

| モックファイル | 移行先 | 用途 |
|--------------|--------|------|
| `src/data/mockUsers.ts`（18名） | Prisma seed スクリプト | 開発・テスト用の初期データ |
| `src/data/mockBands.ts`（7組） | Prisma seed スクリプト | 同上 |
| `src/data/mockEvents.ts`（9件） | Prisma seed スクリプト | 同上 |
| `src/data/mockPosts.ts`（18件） | Prisma seed スクリプト | 同上 |
| `src/data/mockPractice.ts` | Prisma seed スクリプト | 同上 |

```bash
# シードデータの投入（開発環境）
npx prisma db seed

# シードデータのリセット
npx prisma migrate reset
```

### 4.2 本番環境の初期データ

| データ | 内容 |
|--------|------|
| 管理者アカウント | `admin@groovematch.jp`（Cognito + DB） |
| 都道府県・市区町村マスタ | RESAS APIから初回取得→Redisキャッシュ |
| バッジマスタ | 12種のバッジ定義をseedで投入 |
| 招待コード | クローズドベータ用100件を管理画面から生成 |

> テストアカウント（`tanaka@example.com` 等）は本番環境には投入しない。

## 5. localStorage の撤去

全機能のAPI移行が完了した後、以下の手順でlocalStorageへの依存を完全に除去する。

| 手順 | 内容 |
|------|------|
| 1 | フィーチャーフラグを全環境で `false` に設定 |
| 2 | 1週間の安定稼働を確認 |
| 3 | `src/lib/storage.ts` のモック関数を削除 |
| 4 | `src/data/mock*.ts` をテスト専用ディレクトリ（`tests/fixtures/`）に移動 |
| 5 | `NEXT_PUBLIC_USE_MOCK` 環境変数とフィーチャーフラグ分岐コードを削除 |
| 6 | localStorage のキー（`groovematch-*`）を使用するコードが残っていないことを `grep` で確認 |

## 6. 移行チェックリスト

各Wave完了時に確認する。

```markdown
### Wave [N] 移行チェックリスト

- [ ] フィーチャーフラグ `false` でAPI接続が動作する
- [ ] モックデータと同等の操作がAPI経由で可能
- [ ] エラーハンドリング（API障害時のユーザー向けエラー表示）
- [ ] ローディング状態の表示（API応答待ち中のスピナー等）
- [ ] E2Eテストがパスする
- [ ] staging環境で手動確認完了
- [ ] パフォーマンス（API応答時間がP95 < 500ms）
```
