# Groove Match - バックエンド設計

## 概要

Groove Matchのバックエンドは、Next.js API Routes / Server Actions をAPIレイヤーとし、RDS PostgreSQL をメインDB、Amazon Cognito で認証、Spotify API / MusicBrainz で曲データを取得する構成です。

> **現在の状態**: フロントエンドはモックデータ + localStorageで動作しています。本ドキュメントは、バックエンド構築時の設計方針です。

---

## 技術スタック

| 技術 | 用途 | 補足説明 |
|------|------|---------|
| Next.js API Routes / Server Actions | APIエンドポイント | Next.jsに組み込まれたサーバーサイド機能。別途バックエンドサーバーを立てる必要がない |
| Prisma | ORM（DB操作） | データベースをTypeScriptのコードで操作できるツール。SQLを直接書かなくてよい |
| RDS PostgreSQL | メインDB | AWSのマネージドデータベースサービス。初期は db.t4g.micro で開始 |
| Amazon Cognito | ユーザー認証 | AWSの認証サービス。ソーシャルログイン（Google/X/Instagram）にも対応 |
| Stripe | 決済処理 | クレジットカード決済を安全に処理するサービス |
| ElastiCache (Redis) | キャッシュ | メモリ上にデータを保持する高速なDB。Spotify APIのレスポンスキャッシュ等に使用 |
| API Gateway WebSocket | リアルタイムチャット | WebSocketプロトコルによるリアルタイム双方向通信。チャット機能の基盤 |
| Lambda | WebSocketハンドラー | WebSocket接続管理、メッセージ送受信、チャットルーム管理をサーバーレスで処理 |

## 認証設計 (Amazon Cognito)

Amazon Cognito（コグニート）は、AWSが提供するユーザー認証サービスです。パスワードの安全な保存、ソーシャルログイン、JWTトークンの発行などを自動で行ってくれます。

### 認証フロー

```
ユーザー → Cognito → JWT トークン発行
                         |
                         v
              API Routes でトークン検証
                         |
                         v
              DB操作（Prisma経由）
```

### 設定項目

| 項目 | 設定 |
|------|------|
| ユーザープール | groovematch-users |
| パスワードポリシー | 8文字以上、英字+数字必須 |
| MFA（多要素認証） | 任意（将来対応） |
| ソーシャルログイン | Google / X / Instagram（Apple連携は将来対応予定） |
| JWTトークン有効期限 | アクセストークン: 24時間、リフレッシュトークン: 30日 |
| 月間ユーザー数（MAU）1万人まで | 無料（2024年11月以降の新料金体系） |

## 都道府県・市区町村データ

プロフィールの活動地域（F-13）やマッチングの地域スコアで使用する都道府県・市区町村データの管理方針です。

### データソース

都道府県・市区町村データは**外部API**から取得する方式を推奨する。

| 方式 | API / データソース | 特徴 |
|------|-------------------|------|
| **推奨: RESAS API** | 内閣府 地域経済分析システム | 無料。都道府県・市区町村の一覧を取得可能。APIキー必要（無料登録） |
| 代替1: 国土交通省 位置参照情報API | 国土交通省 | 無料。住所データを提供。レスポンスが遅い場合あり |
| 代替2: 静的JSON | 総務省の全国地方公共団体コード（JIS X 0401/0402） | APIなし。JSONファイルとして同梱。市町村合併時に手動更新が必要 |

### 推奨方式: RESAS API + Redisキャッシュ

```
初回起動 / キャッシュ期限切れ時:
  1. RESAS API（api.resas.go.jp）から都道府県一覧を取得
  2. 都道府県ごとに市区町村一覧を取得
  3. Redis にキャッシュ（TTL: 30日）

フロントエンド:
  1. /api/prefectures → 47都道府県を返す（Redisキャッシュ）
  2. /api/prefectures/{code}/cities → 対応する市区町村を返す
  3. 選択した都道府県コード・市区町村コードをDBに保存

バックエンド（マッチング時）:
  1. users テーブルの prefecture / city カラムを比較
  2. 同一市区町村 → 40pt、同一都道府県 → 25pt
```

> **フォールバック**: RESAS APIが利用できない場合に備え、静的JSONデータも同梱する。APIレスポンスをRedisにキャッシュすることで、API障害時もキャッシュから提供できる。

### テーブル設計

`users` テーブルに以下のカラムを持つ:

| カラム | 型 | 説明 |
|--------|-----|------|
| prefecture | VARCHAR(10) | 都道府県コード（例: "13" = 東京都）。JIS X 0401準拠 |
| city | VARCHAR(10) | 市区町村コード（例: "13101" = 千代田区）。JIS X 0402準拠 |
| prefecture_name | VARCHAR(10) | 都道府県名（表示用。例: "東京都"） |
| city_name | VARCHAR(20) | 市区町村名（表示用。例: "千代田区"） |

> 都道府県・市区町村コードを使うことで、表記ゆれ（「東京」「東京都」等）を防ぎ、マッチング時の比較を確実にする。

## データベース設計 (RDS PostgreSQL)

### DB選定理由

| 項目 | 説明 |
|------|------|
| **初期**: RDS PostgreSQL (db.t4g.micro) | 初期フェーズでAurora Serverless v2は過剰。RDS PostgreSQLで**月約¥10,000のコスト削減**が可能 |
| **移行タイミング**: ユーザー数5,000超 | ユーザー数が5,000を超えたらAurora PostgreSQLに移行 |
| **移行コスト**: 最小限 | AuroraもPostgreSQLエンジンなので、Prisma含めて移行コストは最小限。スナップショットからの復元で移行可能 |

### ORM: Prisma

Prisma（プリズマ）は、TypeScriptでデータベースを操作するためのORM（Object-Relational Mapping）ツールです。SQLを直接書く代わりに、TypeScriptのコードでデータの取得・作成・更新・削除ができます。

```typescript
// 例: ユーザーを取得するPrismaのコード
const user = await prisma.user.findUnique({
  where: { email: "tanaka@example.com" },
  include: { instruments: true, genres: true }
});
```

## 外部API連携 (Spotify API / MusicBrainz)

### Spotify API

Spotify API は、Spotifyの楽曲データベースを検索・取得できるAPIです。

| 用途 | エンドポイント | 説明 |
|------|---------------|------|
| 曲検索 | GET /v1/search?type=track | ユーザーが曲を登録するときに検索 |
| 曲情報取得 | GET /v1/tracks/{id} | 曲名・アーティスト名・ジャケット画像の取得 |
| アーティスト情報 | GET /v1/artists/{id} | アーティストのメタ情報取得 |

### MusicBrainz

MusicBrainz は、オープンソースの音楽データベースです。Spotify APIの補完として、Spotifyに登録されていない曲のデータ取得に使用します。

### データフロー

```
ユーザーが曲を登録:
  1. フロントエンドで曲名を入力
  2. Spotify API で検索（リアルタイムサジェスト）
  3. ユーザーが曲を選択
  4. Spotify の track ID だけを DB に保存
     （曲名・アーティスト名などはDBに保存しない）

画面に曲を表示:
  1. DB から Spotify track ID を取得
  2. Redis キャッシュを確認
     - キャッシュあり → キャッシュから返す
     - キャッシュなし → Spotify API から取得 → Redis に保存
  3. 曲名・アーティスト名・ジャケット画像を表示
```

### キャッシュ戦略 (Redis)

| データ | TTL（有効期限） | 理由 |
|--------|---------------|------|
| 曲情報 | 24時間 | 曲のメタ情報は頻繁に変わらない |
| アーティスト情報 | 24時間 | 同上 |
| 検索結果 | 1時間 | 検索結果は新しい曲が追加される可能性がある |
| マッチング結果 | 30分 | ユーザーの設定変更を反映するため短めに設定 |

## クレジットカード情報の安全設計（PCI DSS準拠）

Groove Matchでは月額サブスクリプション（¥500/月）の決済にクレジットカード情報を扱います。**カード情報の漏洩は絶対に防がなければならない**ため、以下の設計原則を採用します。

### 最重要原則: カード情報をサーバーに一切保持しない

```
【Groove Matchの決済フロー】

ユーザーがProプラン登録
    ↓
Stripe Checkout（Stripeのホスト画面にリダイレクト）
    ↓
カード番号の入力はStripeのサーバー上で処理
（Groove Matchのサーバーにカード番号は一切送信されない）
    ↓
Stripe がカード情報をトークン化して保管
    ↓
Groove Match には「顧客ID」「サブスクリプションID」のみが返される
    ↓
Webhook でサブスクリプション状態を同期
```

> **PCI DSS（Payment Card Industry Data Security Standard）** とは、クレジットカード業界のセキュリティ基準です。Groove Matchは**SAQ-A**（カード情報を一切扱わない事業者）に該当するため、最も軽い準拠レベルで済みます。

### アーキテクチャ

| レイヤー | 設計 | 理由 |
|---------|------|------|
| **カード入力** | Stripe Checkout（Stripeホスト画面）を使用 | カード番号がGroove Matchのフロントエンドにも一切渡らない。最も安全 |
| **カード保管** | Stripe側でトークン化・暗号化保管 | PCI DSS Level 1認定を受けたStripeに完全委託 |
| **サーバー保持データ** | Stripe顧客ID（`cus_xxx`）、サブスクリプションID（`sub_xxx`）のみ | カード番号の下4桁すらDBに保存しない |
| **定期課金** | Stripe自動請求 → Webhook通知 | サーバーからカード情報を参照する処理が一切不要 |
| **iOS/Android課金** | App Store / Google Play の標準課金 | Apple/Googleがカード情報を管理。アプリは購入トークンのみ受け取る |

### セキュリティ対策の詳細

| 対策 | 実装 | 説明 |
|------|------|------|
| **Stripe Checkout使用** | リダイレクト方式 | カード入力画面はStripeのドメイン（checkout.stripe.com）で提供。XSS等でカード情報を窃取される可能性をゼロにする |
| **Webhook署名検証** | `stripe-signature` ヘッダー検証 | Webhook通知が本当にStripeからのものか、HMAC-SHA256署名で必ず検証する。偽の決済完了通知を防止 |
| **APIキーの分離** | 公開鍵/秘密鍵の使い分け | フロントエンドでは公開鍵（`pk_`）のみ使用。秘密鍵（`sk_`）はサーバーサイド専用でSecrets Managerに保管 |
| **Secrets Manager** | AWS Secrets Manager | Stripe秘密鍵・Webhook署名シークレットを暗号化保管。自動ローテーション有効。環境変数やコードにハードコードしない |
| **ログからの除外** | 構造化ログ設定 | `stripe_customer_id`以外の決済関連情報をログに出力しない。特にリクエストボディの生ログ出力を禁止 |
| **HTTPS必須** | TLS 1.2以上 | 決済関連のAPI通信は全てHTTPS。HTTP通信は一切許可しない |
| **CSPヘッダー** | Content Security Policy | `frame-src`にStripeドメインのみを許可し、不正なiframeによるカード情報窃取を防止 |
| **Dependabot** | GitHub自動脆弱性検出 | Stripeライブラリ含む全依存パッケージの脆弱性を自動検出・PR作成 |

### Stripe Checkout の実装例

```typescript
// サーバーサイド（API Route）
// カード情報は一切触らない。セッション作成のみ。
export async function POST(request: Request) {
  const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);

  const session = await stripe.checkout.sessions.create({
    mode: 'subscription',
    payment_method_types: ['card'],
    line_items: [{
      price: process.env.STRIPE_PRO_PRICE_ID,  // ¥500/月のPrice ID
      quantity: 1,
    }],
    success_url: `${process.env.APP_URL}/subscription/success`,
    cancel_url: `${process.env.APP_URL}/subscription/cancel`,
    customer_email: user.email,
    // ↑ カード番号はここに一切含まれない
  });

  return Response.json({ url: session.url });
}
```

```typescript
// Webhook ハンドラー（必ず署名検証）
export async function POST(request: Request) {
  const body = await request.text();
  const signature = request.headers.get('stripe-signature')!;

  // 署名検証: 偽のWebhookを完全に拒否
  const event = stripe.webhooks.constructEvent(
    body,
    signature,
    process.env.STRIPE_WEBHOOK_SECRET!
  );

  switch (event.type) {
    case 'checkout.session.completed':
      // DBのユーザープランを 'pro' に更新
      // 保存するのは stripe_customer_id と subscription_id のみ
      break;
    case 'customer.subscription.deleted':
      // DBのユーザープランを 'free' に更新
      break;
  }
}
```

### DBに保存するデータ（subscriptionsテーブル）

| カラム | 保存する | 保存しない |
|--------|:-------:|:---------:|
| user_id | ○ | |
| stripe_customer_id（`cus_xxx`） | ○ | |
| stripe_subscription_id（`sub_xxx`） | ○ | |
| plan（`free` / `pro`） | ○ | |
| status（`active` / `canceled` / `past_due`） | ○ | |
| current_period_end | ○ | |
| **カード番号** | | **× 絶対に保存しない** |
| **カード下4桁** | | **× 保存しない** |
| **有効期限** | | **× 保存しない** |
| **セキュリティコード（CVC）** | | **× 保存しない** |

> カードの下4桁や有効期限をユーザーに表示したい場合は、表示時にStripe API（`GET /v1/customers/{id}/payment_methods`）からリアルタイムで取得する。DBには一切キャッシュしない。

### iOS / Android の決済セキュリティ

| プラットフォーム | 決済方式 | カード情報の管理者 |
|----------------|---------|-----------------|
| iOS | App Store In-App Purchase（StoreKit 2） | **Apple** がカード情報を管理。アプリは購入レシート（トークン）のみ受け取る |
| Android | Google Play Billing | **Google** がカード情報を管理。アプリは購入トークンのみ受け取る |

> モバイルアプリ内課金の場合、Apple/Googleが決済を完全に管理するため、Groove Matchのサーバーはカード情報に一切アクセスできない構造です。

### 禁止事項（開発チーム向け）

| 禁止事項 | 理由 |
|---------|------|
| カード番号をログに出力する | PCI DSS違反。漏洩リスク |
| カード情報をDBに保存する | PCI DSS違反。SAQ-Aから逸脱 |
| Stripe秘密鍵をフロントエンドに配置する | 秘密鍵の漏洩→不正決済のリスク |
| Stripe秘密鍵をコードにハードコードする | GitHub流出時に即座に悪用される |
| Webhook署名検証をスキップする | 偽の決済完了通知によるプラン不正変更 |
| HTTP（非HTTPS）で決済APIを呼ぶ | 通信傍受によるトークン漏洩 |
| Stripe Elements（カード入力フォーム埋め込み）を自前実装する | XSS攻撃でカード情報が窃取される可能性。Stripe Checkoutを使う |

## セキュリティ対策

| 対策 | 実装箇所 | 説明 |
|------|----------|------|
| 入力バリデーション | API Routes | サーバーサイドで全入力を検証（不正なデータを弾く） |
| XSS防止 | React JSX | Reactの自動エスケープ（HTMLタグを無害化する機能） |
| CSRFトークン | next.config.ts | CSRF（クロスサイトリクエストフォージェリ = 偽のリクエストを送る攻撃）を防ぐトークン |
| セキュリティヘッダー | next.config.ts | X-Frame-Options, CSP（コンテンツセキュリティポリシー）, X-Content-Type-Options |
| 認証ガード | ProtectedRoute + API Middleware | 未認証ユーザーのアクセスをブロック |
| 管理者権限チェック | API Middleware | 非管理者からの管理APIアクセスをブロック |
| 通信暗号化 | ALB + CloudFront | TLS 1.2以上（HTTPS必須） |
| パスワード保存 | Cognito | bcrypt（パスワードを暗号化するアルゴリズム）で保存 |
| SQLインジェクション対策 | Prisma | パラメータバインディング（ORMが自動で安全なクエリを生成） |
| WAF | CloudFront | AWS WAF（Web Application Firewall）によるボット対策・レートリミット |

## データモデル（TypeScript型定義）

主要なデータモデルの型定義です。Prismaスキーマはこの型定義をベースに作成します。

### User
```typescript
{
  id: string;
  email: string;
  name: string;
  nickname: string;
  avatar: string;
  bio: string;
  dateOfBirth: string;           // 生年月日（年代表示用に使用、非公開）
  prefecture: string;
  city: string;
  instruments: UserInstrument[];
  genres: string[];
  schedule: Schedule[];
  influences: string[];
  wantToPlaySongs: string[];     // Spotify track IDs
  canPlaySongs: string[];        // Spotify track IDs
  favoriteArtists: string[];     // Spotify artist IDs
  practiceStreak: PracticeStreak;
  socialAccounts: SocialAccount[];
  friends: string[];             // friend user IDs（相互承認済み）
  badges: string[];              // badge IDs
  isAdmin: boolean;
  subscription: 'free' | 'pro';
  authProvider: AuthProvider;
  invitationCode: string;        // 自分の招待コード
  createdAt: string;
  lastLoginAt: string;
  isOnline: boolean;
}
```

### Band
```typescript
{
  id: string;
  name: string;
  description: string;
  genre: string[];
  prefecture: string;
  city: string;
  members: BandMember[];
  maxMembers: number;
  instrumentSlots: InstrumentSlot[];
  imageUrl: string;
  isRecruiting: boolean;
  setlist: SetlistItem[];        // Spotify track IDs + ステータス
  targetArtists: string[];       // Spotify artist IDs
  createdAt: string;
  createdBy: string;
}
```

### LiveEvent
```typescript
{
  id: string;
  title: string;
  description: string;
  venue: string;
  address: string;
  prefecture: string;
  date: string;
  startTime: string;
  endTime: string;
  capacity: number;
  registeredBands: string[];
  tags: string[];
  isBeginnerFriendly: boolean;
  fee: number;
  imageUrl: string;
  organizerId: string;
  isOfficial: boolean;           // 運営登録 = true（「公式」バッジ表示）
  createdAt: string;
}
```

その他のデータモデル（Post, PracticeLog, Badge, SocialAccount, AppNotification等）は `src/lib/types.ts` を参照してください。
